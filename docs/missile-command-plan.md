# Missile Command 実装メモ

## 目的
- Rust 上で `Missile Command` ライクな防衛ゲームを再構築する。
- 第1段階は macOS でのプレイ成立を最優先にする。
- 将来的な Windows / Linux 展開を見据え、描画・入力・ウィンドウ制御は単一クレートに寄せる。

## 採用方針
- ランタイムは `macroquad` を採用する。
- 理由:
  - macOS でフルスクリーン、キーボード、マウス入力、2D 描画が単一依存でまとまる。
  - Windows / Linux にも同じコードで持っていきやすい。
  - 今回のゲームはスプライト資産よりも、線・円・簡易 HUD を高速に描けることが重要。

## 第1段階の実装対象
- フルスクリーン起動
- 照準移動
  - カーソルキー
  - マウス移動
- 発射操作
  - `z`: 左基地
  - `x`: 中央基地
  - `c`: 右基地
- 制御
  - `space` / `p`: 一時停止
  - `q`: 強制終了
- ルール
  - 3基地、6都市、防衛対象の消失
  - 敵ミサイルの波状攻撃
  - 自軍ミサイルの着弾爆風による迎撃
  - ウェーブ進行
  - 残弾ボーナス、都市生存ボーナス
  - スコア閾値による都市復旧

## 現在の追加仕様
- タイトル画面
- ゲームオーバー後のリスタート導線
- 生成音ベースの効果音と簡易 BGM
- 設定画面
  - 基地ごとの装弾数
  - 爆風半径
  - 難易度
- スマートボムの部分実装
  - 通常飛行中も軽く蛇行する
  - プレイヤー爆風を検知すると横方向へ回避する
  - アーケード寄せとして後半ウェーブから出現
  - 難易度に応じて出現波と出現率を変える
- 爆撃機
  - 中高度を横断しながら地上へ投弾する
  - 空中支援として1機ずつ順次出現
- 衛星
  - 高高度を高速横断しながら地上へ投弾する
  - 爆撃機より高速で、後半から混在する

## 現在の調整方針
- アーケード寄りの段階感
  - 空中支援は早めのウェーブから
  - スマートボムは後半ウェーブから
- スコア
  - 通常弾 / 分裂弾: 25
  - スマートボム: 125
  - 爆撃機 / 衛星: 100
- 難易度差分
  - 敵速、出現間隔、空中支援数、スマートボム出現波で変化を付ける

## 初版での設計分割
1. `main.rs`
   - ウィンドウ設定
   - メインループ
2. `game.rs`
   - ゲーム状態
   - 入力処理
   - ウェーブ制御
   - ミサイル、爆風、当たり判定
   - 描画

## ルール上の意図的な簡略化
- まずは通常弾道ミサイルと分裂弾を実装対象にする。
- スマートボム、爆撃機、衛星は次段階の追加要素とする。
- ただし、コード構造は敵種追加を前提とする。

## ウェーブ設計
- 各ウェーブで敵数と速度を少しずつ上げる。
- 中央基地は左右基地より高速に発射する。
- ウェーブ終了時に:
  - 未使用ミサイル数に応じたボーナス
  - 生存都市数に応じたボーナス
  - 次ウェーブ開始時に基地弾数を補給

## 今回の構築順
1. 設計メモを残す
2. `macroquad` 導入
3. 画面・入力・HUD を実装
4. ミサイル、爆風、当たり判定を実装
5. ウェーブ進行、スコア、ゲームオーバーを実装
6. `cargo fmt` / `cargo check` で確認

## 次段階の拡張候補
- 爆撃機 / 衛星
- サウンド
- 難易度テーブルの本格調整
- タイトル画面、リスタート導線

## 改善案メモ
- 今回のアーケード寄せ調整分をコミットする
- 実プレイ前提でウェーブ別の出現表をさらに細かく詰める
- タイトル画面に現在設定のプレビューや敵紹介を追加する

## WebAssembly / HTML5 方針
- 段階1: static HTML + WASM + JS バンドル + 任意アセットで完結する構成にする
  - 静的ホスティングにそのまま置けることを優先する
  - ローカル保存は `localStorage` を使い、設定やローカルハイスコアを保持する
  - ランキング機能はまだ載せず、ゲーム本体だけをブラウザで成立させる
- 段階2: オンラインランキング用の API サーバーを追加する
  - ブラウザ版は HTTP API 経由でスコア送信 / 取得
  - macOS / Linux / Windows ネイティブ版も同じ API に参加できる構成を目指す
  - スコア改ざん対策、名前重複、レート制限を別途検討する

## WebAssembly 実装可否メモ
- 現状の `macroquad` 採用は HTML5 / WASM と相性が良く、段階1の実装は可能と判断する
- 想定成果物
  - `index.html`
  - `mq_js_bundle.js`
  - `mcommand.wasm`
  - `assets/` 配下の任意ファイル
- 変更が必要なポイント
  - ネイティブ前提のフルスクリーン挙動をブラウザ前提へ調整する
  - 終了操作 `q` は「ゲーム終了」ではなくタイトルへ戻る、または無効化する方が自然
  - 設定保存とローカルスコア保存をブラウザ永続化へ切り替える
  - 音声の自動再生制約に合わせ、初回ユーザー操作後に BGM を開始する
- 段階2への接続も可能だが、まずは static 配信を成立させてから API を足すのが安全
